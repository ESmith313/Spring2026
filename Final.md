##Library Site Architecture Documentation

 - This project centered around WordPress, Omeka, and Koha. They work together to create a simple but functional library ecosystem. All three systems were installed and configured using Google Cloud virtual machine (VM). The systems are split across two VMs, WordPress and Omeka share one while Koha runs on its own virtual machine. This setup helps balance organization and performance while keeping each system manageable.
 - WordPress acts as the main entry point for the entire site. It’s the first thing users see and function as the public facing website. Realistically this would include general information like services, announcements, and basic pages. More importantly, WordPress serves as the central navigation hub. At the top of the site, there are links to both the Koha OPAC and the Omeka site, including both their public and admin views. By placing these links in a consistent location, WordPress connects all parts of the system and makes it easy for users to move between them.
 - Omeka is hosted on the same VM as WordPress, both are more content-focused systems. Omeka is used for digital collections and exhibits, such as images, documents, or curated projects. The public side allows users to browse these collections, while the admin side is used to upload items, add metadata, and organize exhibits. Since it shares a VM with WordPress, the connection between the two feels more direct, and it simplifies server management by grouping similar types of content together.
 - Koha on the other hand is hosted on its own separate VM. This is purposeful because Koha is a more complex system that handles core library functions like cataloging, circulation, and patron accounts. By isolating it on its own server, it can run more efficiently and avoid potential conflicts with the other platforms. From the user’s perspective, Koha is accessed through a link on the WordPress site, so even though it is technically separate, it still feels like part of the overall system. The admin side of Koha is used by staff to manage records and users, while the public interface allows patrons to search the catalog and access their accounts.
 - Navigating across the system is kept simple on purpose. Instead of trying to merge everything into one interface, the design relies on clear and direct links. The top menu on WordPress is the main tool for guiding users. If someone wants to search for books, they go to Koha. If they want to explore digital exhibits, they go to Omeka. This approach reduces confusion and makes it obvious where each type of content lives.
 - The decision to split the systems across to VMs is also part of the overall architecture. Keeping WordPress and Omeka together works well because they are both lightweight compared to Koha and focus on content delivery. Separating Koha allows it to use its own resources and improves stability, especially since it handles more demanding tasks. This structure also makes troubleshooting easier. If one system has an issue, it doesn’t necessarily affect the others.
 - This setup reflects how many real-world libraries organize their systems. It’s common to have a main website, a catalog system, and a digital collections platform, often hosted separately but connected through navigation. While larger institutions might use more advanced integrations, this approach still follows the same basic idea of using specialized tools for different purposes.
 - The system works because each part has a clear role and the connections between them are straightforward. WordPress acts as the front door, Omeka provides access to digital collections, and Koha manages the catalog. The use of two Google Cloud VMs supports this structure by balancing efficiency and simplicity. By focusing on clear responsibilities and intuitive navigation, the system ensures that users can move easily between platforms without confusion, creating a cohesive and user-friendly library experience.
